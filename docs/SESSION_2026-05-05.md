## 1. Übersicht

Ziel der Session: das bisher rein manuell verifizierte Projekt mit einer kompletten **Test-Pyramide** (Unit / Integration / E2E) ausstatten und in eine **GitHub-Actions-Pipeline** integrieren — als Vorbereitung für den Live-Rollout beim Kunden.

Außerdem wurden strukturelle Verbesserungen vorgenommen, die durch die Test-Anforderungen sichtbar wurden: PDF-Extraktor in eigenen Service ausgelagert, App-Factory von Server-Bootstrap getrennt, `uniquifyName` exportierbar gemacht.

**Test-Strategie (vom User bestätigt):**
- Unit + Integration: jeder Push, jeder PR
- E2E: nur `main` + Pull-Requests (langsamer, weniger oft)

**Stand am Sessionende:** 58 Tests grün, CI-Workflow live, Stack rebuild sauber.

## 4. Phase C — Backend Integration-Tests (5 Tests)

### Refactor: App-Factory
[server/src/app.ts](../server/src/app.ts) neu — exportiert `createApp()`. [server/src/index.ts](../server/src/index.ts) bleibt minimal: Bootstrap (Migrate, Upload-Dirs anlegen) + `app.listen()`. So kann `supertest` direkt gegen die App-Instanz testen, ohne den Server auf einem Port zu starten.

### Setup-Infrastruktur
- [server/src/test/integration-setup.ts](../server/src/test/integration-setup.ts) — setzt vor Modul-Imports alle ENV-Variablen (Test-DB, JWT-Secrets, FTP-Defaults)
- [server/src/test/integration-helpers.ts](../server/src/test/integration-helpers.ts) — `setupTestDb()`, `truncateAllTables()`, `seedAdminUser()`, `seedBauleiterUser()`, `getTestApp()`

### Test-Suite
[server/src/routes/auth.integration.test.ts](../server/src/routes/auth.integration.test.ts) — 5 Tests gegen den Auth-Flow:
- Login mit korrekten Credentials → Token + HTTP-only-Cookie
- Login mit falschem Passwort → 401
- Login mit unbekanntem User → 401
- `GET /me` mit gültigem Token → User-Info
- `GET /me` ohne Token → 401

### Test-DB
Lokal: separate Datenbank `bautagesbericht_test` auf der bestehenden Postgres-Instanz, einmalig anzulegen:
```bash
docker compose exec postgres psql -U patzig -d postgres -c "CREATE DATABASE bautagesbericht_test;"
```
In CI: GitHub-Actions Service-Container erzeugt die DB automatisch.

## 7. Phase F — GitHub Actions Pipeline

[.github/workflows/test.yml](../.github/workflows/test.yml) mit 5 Jobs:

| Job | Trigger | Beschreibung |
|---|---|---|
| `lint-typecheck` | jeder Push/PR | TypeScript-Check für Server + Client |
| `backend-unit` | jeder Push/PR | Vitest Unit-Tests + Coverage als Artifact |
| `backend-integration` | jeder Push/PR | Postgres als Service-Container, Integration-Suite |
| `frontend-unit` | jeder Push/PR | Vitest Frontend-Tests + Coverage |
| `e2e` | nur `main` + PRs | Hochfährt `docker-compose.test.yml`, Playwright-Tests, Reports bei Fehler als Artifact |

Bestehender [docker-build.yml](../.github/workflows/docker-build.yml) bleibt unverändert (Deploy nach GHCR bei Push auf `main`).
