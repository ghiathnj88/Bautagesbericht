## 2. FTP-Browser für Arbeitsauftrag-Import

### Problem
Bisher musste der Bauleiter das Arbeitsauftrag-PDF von seinem Gerät hochladen. Die Kunden-PDFs liegen aber bereits auf dem FTP-Server der Firma — ein doppelter Weg, der Fehler begünstigt.

### Lösung
Neuer Modal-Picker, der direkt das FTP-Verzeichnis des Bauleiter-Users durchsucht und PDFs zur Auswahl anbietet. Der Picker startet automatisch im Home-Verzeichnis des FTP-Users (z.B. `/ftp/berichte` bei Patzig) und erlaubt Navigation in Unterordner.

### Änderungen
- **[server/src/services/ftp.service.ts](../server/src/services/ftp.service.ts)** — neue Funktionen:
  - `listFtpDir(remoteDir?)` — listet Ordner + PDFs, nutzt `pwd()` wenn kein Pfad angegeben
  - `downloadFromFtp(remotePath)` — lädt Datei via Stream in einen Buffer
  - `FtpListResult` / `FtpEntry` Types
- **[server/src/routes/report.routes.ts](../server/src/routes/report.routes.ts)** — zwei neue Routen:
  - `GET /api/reports/ftp-browse?path=…` — gibt `{path, entries}` zurück
  - `POST /api/reports/extract-pdf-ftp` — lädt PDF vom FTP, parst via `pdf-parse`, extrahiert Felder
  - Path-Traversal-Schutz (kein `..`, nur `.pdf`-Dateien)
- **[client/src/components/FtpPdfPicker.tsx](../client/src/components/FtpPdfPicker.tsx)** — neues Modal (Ordner-Navigation, Breadcrumb, Zurück-Button, Lade-/Fehlerzustand)
- **[client/src/components/ReportForm.tsx](../client/src/components/ReportForm.tsx)** — alter Device-Upload komplett entfernt, durch FTP-Picker-Button ersetzt

### FTP-User-Home-Verzeichnis
Der delfer/alpine-ftp-server chrootet den Nutzer nicht. Beim ersten Call ohne `path`-Parameter wird `client.pwd()` aufgerufen, was den Login-Home-Pfad zurückgibt (`/ftp/berichte` bei Patzig). Von da aus kann der Bauleiter navigieren.

### Docker Compose env vars
In beide Compose-Dateien ergänzt:
```yaml
FTP_HOST: ${FTP_HOST}
FTP_PORT: ${FTP_PORT:-21}
FTP_USER: ${FTP_USER}
FTP_PASSWORD: ${FTP_PASSWORD}
```
Lokale `.env` bekommt die Werte für den Hostinger-FTP-Server (`72.60.178.75`, `testuser`/`testpass`).

## 3. Patzig-spezifische PDF-Extraktion

### Problem
Das erste Test-PDF (`Arbeitsauftrag_200003563.pdf`) konnte nicht korrekt geparst werden: `auftraggeber=""`, `lieferanschrift=""`, `bvNummer="."`, `kundennummer="."`.

### Ursache
`pdf-parse` zerlegt die Tabelle aus dem PDF-Layout zeilenlinear, wodurch Werte und Labels auseinandergerissen werden:
```
: 200003563        ← Nummer-Wert
: 201487489        ← Projektnr-Wert
Projektnr.         ← Labels kommen DANACH
Kundennr.
Datum
Nummer
Original
Objekt 24046 / WEG Guttenbrunnstr. 28-36 +
…
: 08.09.2025       ← Datum-Wert
: 201313552        ← Kundennr-Wert
```
Die generische "Label-auf-Zeile"-Regex fand so keine Zuordnung.

### Lösung
Neue Funktion `extractPatzigArbeitsauftrag()` in [report.routes.ts](../server/src/routes/report.routes.ts), die:

1. Das Format an der Zeile `"Arbeitsauftrag"` erkennt
2. Alle Zeilen sammelt, die mit `: ` beginnen (= Tabellenwerte aus der rechten Spalte)
3. Die Datumszeile (`DD.MM.YYYY`) als Anker findet
4. **Vor** dem Datum: 2. Zahlenwert = Projektnr (BV-Nummer)
5. **Nach** dem Datum: 1. Zahlenwert = Kundennr
6. Adressblock zwischen `"Original"` und der ersten `:`-Wertzeile liest
7. Bei `"c/o "` splittet: davor = Lieferanschrift (Baustelle), danach = Auftraggeber (Rechnungsempfänger)

Generische Label-Logik bleibt als **Fallback** für andere PDF-Formate.

### Lösung

## 5. Foto-Layout im PDF

### Änderungen in [server/src/services/pdf.service.ts](../server/src/services/pdf.service.ts)

**Alt:** Fotos mitten im Bericht (zwischen Entsorgung und Wetter), je 160×120 px klein, Flexbox-wrap.

**Neu:**
- Fotos werden ans **Ende des Berichts verschoben**, direkt vor die Unterschriften
- 2-Spalten-CSS-Grid, je Foto 100% breit × 80 mm hoch, `object-fit: cover`
- **Maximal 6 Fotos pro Seite** (3 Reihen × 2 Spalten)
- Fotos werden in JS in Gruppen von 6 geteilt
- Jede Gruppe ist ein `.photo-page-group` mit `page-break-inside: avoid` + `page-break-after: always`

## 6. Bericht-Auslieferung: E-Mail + FTP im Quell-Verzeichnis

### Anforderung
Wenn der Bericht abgesendet wird, soll er:
1. Als PDF per E-Mail an den Kunden gehen (wie bisher vorgesehen)
2. **Zusätzlich** im FTP-Server im **gleichen Verzeichnis** gespeichert werden, aus dem der Arbeitsauftrag importiert wurde — nicht mehr im hardcoded `berichte/`-Ordner

### Lösung
- **Neues Feld `ftpSourcePath: string`** in [ReportData](../client/src/types/report.ts)
- In `handleFtpPdfSelect` in [ReportForm.tsx](../client/src/components/ReportForm.tsx) wird der **Parent-Ordner** des ausgewählten PDFs automatisch in `data.ftpSourcePath` gespeichert:
  ```ts
  const parentDir = remotePath.substring(0, remotePath.lastIndexOf('/')) || '/';
  ```
- In [report.routes.ts](../server/src/routes/report.routes.ts) wurde `autoUploadToFtp` umbenannt zu **`deliverCompletedReport`** und um zwei Dinge erweitert:
  1. **FTP-Upload**: `${dataJson.ftpSourcePath}/Bautagesbericht_${bv}_${datum}.pdf` (mit Fallback `berichte/${fileName}` wenn kein Source-Path gesetzt)
  2. **E-Mail-Versand**: ruft `sendReportEmail` aus dem existierenden Email-Service mit dem generierten PDF als Anhang
- Beide Kanäle sind in eigenen `try/catch`-Blöcken → **ein Fehler in einem blockiert den anderen nicht**
- Logging-Präfixe: `[FTP] …`, `[Email] …`

### Beispiel-Flow
1. Bauleiter öffnet Picker → wählt `/ftp/berichte/berichte/Arbeitsauftrag_200003563.pdf`
2. Form bekommt:
   - `ftpSourcePath = "/ftp/berichte/berichte"`
   - extrahierte Felder (Auftraggeber, Lieferanschrift, BV-Nr, Kundennr)
3. Bauleiter füllt aus, klickt "Senden"
4. Server generiert PDF und:
   - lädt hoch nach `/ftp/berichte/berichte/Bautagesbericht_201487489_13-04-2026.pdf`
   - versendet PDF per E-Mail an `customerEmail` (falls SMTP konfiguriert)

### SMTP
Der Email-Service liest SMTP-Einstellungen aus der verschlüsselten `settings`-Tabelle. Lokal aktuell **nicht konfiguriert** → Logzeile `[Email] Versand fehlgeschlagen: SMTP-Einstellungen nicht konfiguriert`. Der FTP-Upload läuft trotzdem durch. Für Live-Betrieb: SMTP im Admin-Panel konfigurieren.

## 7. Bugfix: Datum aus altem Draft

### Problem
Nach dem ersten Test-Submit hatte der gespeicherte Bericht das Datum `01.04.2026` im Dateinamen, obwohl er am 13.04.2026 erstellt wurde.

### Ursache
Das Formular speichert den Entwurf in `localStorage` unter `bautagesbericht_draft`. Beim Laden wurde der gesamte Draft inklusive altem `datum`-Feld wiederhergestellt. Da der initiale Entwurf am 01.04. begonnen wurde, blieb `datum: "01.04.2026"` erhalten.

### Lösung
In [ReportForm.tsx](../client/src/components/ReportForm.tsx) in der Funktion `loadDraft()`:
```ts
const today = new Date().toLocaleDateString('de-DE', { day: '2-digit', month: '2-digit', year: 'numeric' });
if (raw) return { ...createEmptyReport(), ...JSON.parse(raw), datum: today };
```
Der `datum`-Override greift nach dem Spread des Drafts, womit das Datum **immer auf heute gesetzt** wird. Falls ein Bauleiter explizit ein anderes Datum (z.B. für einen Nachtrag) braucht, kann er das Feld weiterhin manuell ändern.

## 8. Bugfix: Logout bei Page-Refresh

### Problem
Bei jedem `F5`/Browser-Refresh wurde der Nutzer ausgeloggt und musste sich neu anmelden.

### Ursache
In [client/src/auth/AuthContext.tsx](../client/src/auth/AuthContext.tsx) wurde der Refresh-Endpoint falsch aufgerufen:
```ts
const data = await apiFetch<{ accessToken: string }>('/auth/refresh');
```
`apiFetch` ohne `method`-Option defaultet zu **GET**. Der Server hat aber nur `POST /auth/refresh`:
```
GET  /api/auth/refresh → 404
POST /api/auth/refresh → 401 (ohne Cookie) / 200 (mit Cookie)
```
Folge: jeder Page-Load scheiterte mit 404, AuthContext interpretierte das als "nicht eingeloggt", setzte User auf `null`.

### Lösung
Einzeilen-Fix in [AuthContext.tsx](../client/src/auth/AuthContext.tsx):
```ts
const data = await apiFetch<{ accessToken: string }>('/auth/refresh', { method: 'POST' });
```
