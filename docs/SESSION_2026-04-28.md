## 5. Bericht- und Foto-Pfade auf Patzig-Konvention

[server/src/routes/report.routes.ts](../server/src/routes/report.routes.ts) `deliverCompletedReport()`:

- `monteurDir` wird aus `ftpSourcePath` extrahiert (= Parent-Ordner des Quell-PDFs, also der Monteur-Ordner)
- **BTB-PDF** landet in `${monteurDir}/BTB/Bautagesbericht_<bv>_<datum>_<shortId>.pdf`
- **Fotos** landen in `${monteurDir}/<YY-MM-DD>/<shortId>_<dateiname>.jpg`

Neue Hilfsfunktion `formatYyMmDd(datum)`: wandelt `DD.MM.YYYY` (Wizard-Format) in `YY-MM-DD`-Ordnername um.

`uploadToFtp` legt fehlende Unterordner per `client.ensureDir()` automatisch an.

Auch die manuelle `/ftp-upload`-Fallback-Route nutzt das neue Schema mit shortId.

### shortId-Mechanik (Schutz vor Kollisionen)

Wenn am gleichen Tag im gleichen Auftrag **zwei Berichte** entstehen (Früh-/Spätschicht oder zwei Bauleiter parallel), würden sich Bericht-Dateinamen und Foto-Namen überschreiben. Lösung: **8-stellige Hex-Kennung** (erste 8 Zeichen der Bericht-UUID ohne Bindestriche) wird:

- an den **PDF-Dateinamen** angehängt: `Bautagesbericht_201487489_28-04-2026_a1b2c3d4.pdf`
- als **Präfix** vor jeden Foto-Dateinamen gesetzt: `a1b2c3d4_IMG_001.jpg`

Beide Berichte teilen sich denselben Datums-Ordner — die Bilder sind über den Präfix eindeutig dem jeweiligen Bericht zuordenbar. PDF und Fotos eines Berichts haben dieselbe shortId.

Kollisionswahrscheinlichkeit bei 32-bit Hex-Präfix: praktisch null (≈ 4 Mrd Möglichkeiten).

## 8. Foto-/Signatur-/PDF-Cleanup beim Bericht-Löschen

[server/src/routes/report.routes.ts](../server/src/routes/report.routes.ts) `DELETE /:id`:

- Vor dem `db.delete(reports)` werden die zugehörigen `report_photos`- und `report_signatures`-Einträge geladen
- Alle physischen Dateien (`uploads/photos/*`, `uploads/signatures/*`, `uploads/pdfs/*`) werden via `fs.unlink` entfernt
- Fehler beim Unlink werden toleriert (Logging, kein Abbruch)
- Anschließend DB-Delete; FK CASCADE räumt die Einträge in `report_photos`/`report_signatures` automatisch
- Response: `{ ok: true, filesRemoved: <count> }`

Damit verbleiben **keine Waisen-Dateien** im uploads-Volume.

## 12. Manueller Upload von Bildern/Videos

### Anforderung (Patrick-Mail vom 27.04.2026)
> *"Der Prozess passt so aber es wäre super wenn es im Nachhinein auch möglich ist, weitere Bilder und Videos separat in den Ordner hochzuladen."*

### Backend

[server/src/services/ftp.service.ts](../server/src/services/ftp.service.ts):
- Neue Funktion `uploadBufferToFtp(buffer, remotePath)`: streamt einen In-Memory-Buffer via `Readable.from(buffer)` direkt zum FTP, mit `ensureDir`

[server/src/routes/report.routes.ts](../server/src/routes/report.routes.ts):
- Neue Multer-Konfiguration `adhocUpload`: memoryStorage, Filter auf `image/*`/`video/*`, 200 MB pro Datei
- Neue Route `POST /reports/ftp-upload-files`:
  - `multer.array('files', 20)` — max. 20 Dateien gleichzeitig
  - Pflicht-Body-Feld `remoteDir`
  - Pfad-Validierung: kein `..`, normalisiert
  - Schleife: pro Datei `uploadBufferToFtp(buffer, remoteDir + '/' + safeName)` mit `path.posix.basename` zur Sanitisierung
  - Response: `{ uploaded: string[], failed: { name, error }[], remoteDir }`
  - Ein Fehler stoppt den Rest nicht
- Authentifiziert via bestehende Token-Middleware

### Frontend

[client/src/components/FtpBrowser.tsx](../client/src/components/FtpBrowser.tsx) erweitert:

- Hilfsfunktionen `todayYyMmDd()`, `isoToYyMmDd(iso)`, `yyMmDdToIso(yy)` für Datums-Konvertierung
- `canUpload`-Flag: nur sichtbar wenn `path.includes('/Monteur')`
- **„+ Bilder/Videos"-Button** rechts in der Breadcrumb-Leiste
- **Im Monteur-Ordner**: Klick öffnet Datums-Auswahl-Dialog (`<input type="date">`, Default: heute) mit Pfad-Vorschau
- **In einem Datums-Ordner**: Klick öffnet direkt File-Picker (Datum aus aktuellem Pfad)
- File-Picker: `accept="image/*,video/*"`, `multiple`, versteckt
- Bei Auswahl: FormData mit `files[]` und `remoteDir` → POST an Backend
- Loading-Banner blau („Dateien werden hochgeladen…")
- Status-Banner grün/gelb mit Erfolgs-/Fehler-Aufzählung
- `loadDir(path)` nach Upload → Liste lädt neu

`apiFetch` aus [client/src/api/client.ts](../client/src/api/client.ts) lässt Content-Type bei FormData automatisch weg, sodass der Browser die Boundary korrekt setzt.

### Verhalten bei manuellem Upload + späterem BTB

Wenn Bilder zuerst manuell und später per BTB im selben Datum hochgeladen werden:

```
Monteur/26-04-28/
  IMG_morgens.jpg              ← manuell, kein Präfix
  Schaden_links.jpg            ← manuell, kein Präfix
  a1b2c3d4_IMG_001.jpg         ← BTB-Foto, mit shortId-Präfix
  a1b2c3d4_IMG_002.jpg         ← BTB-Foto
```

`ensureDir` erkennt den vorhandenen Ordner und legt nicht neu an. Manuell und BTB können nicht kollidieren (verschiedene Namens-Schemas).

**Bekannter Edge-Case:** zwei manuelle Uploads mit identischem Dateinamen würden sich gegenseitig überschreiben. Dem Kunden ist das Risiko bewusst gemacht; eine Mitigation (Zeitstempel-Präfix oder Auto-Suffix) kann später ergänzt werden, falls relevant.
